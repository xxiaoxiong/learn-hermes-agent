# h03a — Agent-level Tools：为什么有些工具不该走普通注册表

> 这页是 `h03` 的延伸。主线章节告诉你 `todo` 会在主循环里被优先拦截；这份 bridge doc 要把这个设计再说透：**为什么有些工具不只是“能调用的能力”，而是 agent 自身控制面的组成部分。**

---

## 先说结论

不是所有 tool 都属于同一层。

Hermes 里至少有两类：

- **普通工具**：作用到外部世界，例如文件、shell、网络
- **agent-level tools**：作用到 agent 自己的内部状态、记忆或历史检索能力

如果把后一类也强行塞进普通 `ToolRegistry.dispatch()`，系统表面上会更“统一”，但控制边界会变糊，agent 对自身状态的管理也会退化。

---

## 1. 普通工具改外部世界，agent-level tools 改内部工作台

先看最直观的区别：

| 类型 | 典型例子 | 改动对象 |
|---|---|---|
| 普通工具 | `read_file`, `write_file`, `bash` | 文件系统、操作系统、网络 |
| agent-level tools | `todo`, `session_search`, `memory_write` | agent 的计划、历史记忆、内部检索上下文 |

这条边界很重要。

因为 `read_file` 失败时，影响的是外部任务推进；但 `todo` 失败时，影响的是 agent 对自己“做到哪了”的认知。后者不是普通业务动作，而是控制面的一部分。

---

## 2. 为什么 `todo` 最容易看懂这件事

`todo` 的本质不是“往外界写一条记录”，而是：

- 建立步骤列表
- 更新当前执行状态
- 维持任务的过程性记忆

也就是说，`todo` 操作的是 `PlanState`。而 `PlanState` 又直接影响：

- 模型下一步怎么判断优先级
- 当前任务是否已经完成
- 复杂目标是否需要继续拆解

如果你把它做成一个普通 handler，它当然也能跑，但语义上会变成：

> agent 去调用一个外部服务，问它“我的计划现在长什么样？”

这就弱化了 agent 的自我状态管理能力。

---

## 3. `session_search` 为什么也更像 agent-level tool

很多人第一次看到 `session_search` 会误以为它只是一个普通查询工具。

但它和 `read_file` 不同的地方在于：

- 它不是去读一个外部业务资源
- 它读的是 agent 自己过去的会话历史
- 返回的内容通常会重新进入当前 `messages`，影响后续推理

所以它更像：

**agent 访问自己的长期工作记忆。**

这就是为什么它和 `todo` 虽然表面用途不同，但在架构上有同一类味道：都属于 agent 内部工作台能力，而不只是外界 I/O。

---

## 4. `memory_write` 也是同一类问题

`memory_write` 表面上像“把一段文本写入文件”。

但在 Hermes 的语境里，它做的其实不是普通文件写入，而是：

- 把上下文中值得长期保留的知识筛出来
- 写入 `MEMORY.md` / `USER.md`
- 改变未来会话的 prompt 输入

换句话说，`memory_write` 影响的是：

- agent 以后会记住什么
- system prompt 里会注入什么长期知识
- 多轮、多会话之后的行为稳定性

它影响的已经不是当前回合某个外部动作，而是 agent 的长期认知状态。

---

## 5. 为什么“统一走注册表”听起来更优雅，却不一定更对

很多架构设计初看会追求一个目标：

> 所有工具都统一走一条路径，最整齐。

这个想法没错，但要看你统一的是什么。

如果统一的是：

- 模型看到的工具说明
- tool call 的解析格式
- 返回结果的消息形状

这通常是好事。

但如果统一到连**控制面和执行面都不再区分**，问题就来了：

- 计划状态会被误当成外部副作用
- 历史检索会被误当成普通资源查询
- 记忆写入会被误当成普通文件操作

结果就是系统表面更整齐，实际边界更模糊。

---

## 6. 判断一个工具是不是 agent-level，可以问这三个问题

### 问题 1：它主要改的是外部世界，还是 agent 自己的状态？

- 改外部世界，更像普通工具
- 改内部状态，更像 agent-level tool

### 问题 2：它的结果主要给用户看，还是主要为了 agent 下一步继续推理？

- 给用户看，更像普通工具
- 给 agent 自己继续推理，更像 agent-level tool

### 问题 3：如果它失败了，损失的是业务动作，还是 agent 的控制能力？

- 业务动作失败，更像普通工具
- 控制能力受损，更像 agent-level tool

`todo`、`session_search`、`memory_write` 基本都更偏第二种答案。

> 这些看起来都叫 tool，但它们在架构层里并不属于同一层。

---

## 8. 这页最该带走的一句话

如果一个 tool 改变的是 **agent 自己的计划、记忆或历史访问能力**，那它通常就不只是普通工具，而更接近 agent 的控制面能力。

这也是 Hermes 为什么要在某些地方故意保留“主循环先拦截，再决定是否走 registry”这一层结构。
