# h06a — Session Search：为什么检索历史会话不只是“数据库查询”

> 这页是 `h06` 的延伸。主线章节讲了 SQLite + FTS5 如何保存和搜索 session；这份 bridge doc 要强调另一层关键理解：**当 `session_search` 被 agent 主动调用时，它不再只是数据层查询，而变成了 agent 的内部回忆机制。**

---

## 先说结论

如果从数据库视角看，`session_search` 只是：

- 给一个 query
- 返回命中的历史记录

但如果从 agent 视角看，它实际上是在做：

- 回忆过去是否出现过类似问题
- 检索某段历史上下文
- 把历史线索重新带回当前推理链

所以 `session_search` 的意义，远远不只是“数据库里有个 search 方法”。它是 session storage 真正接入 agent loop 的桥。

---

## 1. 为什么单有 SQLite 还不够

把 session 保存下来，只解决了第一步：

- 历史没有丢

但保存不等于可用。

如果 agent 无法在需要时主动把历史捞回来，那么这些 session 更像冷存档，而不是可工作的记忆系统。

所以 `h06` 真正要落地，不只是：

- 有一个 `save_session()`
- 有一个 `search()`

而是：

- agent 能在对话过程中主动触发 `session_search`
- 搜索结果能重新进入当前上下文
- 模型可以基于历史线索继续推理

---

## 2. `session_search` 为什么和普通搜索工具不一样

表面上看，它和搜索文件、搜索网页很像，都是：

- 输入关键词
- 返回命中结果

但它和普通搜索工具最大的不同是：

- 搜的不是外部资料
- 搜的是 agent 自己过去的会话历史
- 返回结果往往和当前任务具有时间连续性

也就是说，`session_search` 更像：

**agent 对自己工作记忆的一次内部检索。**

这也是为什么它更接近 agent-level tool，而不只是普通资料搜索。

---

## 3. 从“存档”到“回忆”，中间差了一次注入

理解 `session_search` 最关键的一点是：

数据库命中结果本身，不会自动改变模型行为。

真正让它变成“回忆机制”的，是下面这一步：

- 检索结果被注入当前 messages
- 模型在下一轮看见这些历史摘要或片段
- 当前推理因此得到过去经验的补充

也就是说，回忆不是发生在数据库里，而是发生在：

**检索结果重新进入上下文之后。**

---

## 4. 为什么 FTS5 对 agent 来说不是实现细节

对数据库工程师而言，FTS5 是全文索引技术。

但对 agent 来说，它带来的不是一个 SQL 特性，而是一个能力边界：

- 能不能在大量历史中快速定位相关会话
- 能不能在上下文已经压缩后，重新找回旧线索
- 能不能让“过去做过类似任务”真正影响当前回合

如果没有全文索引，session 虽然存下来了，但检索会过慢、过脆弱，最后就不太可能成为 agent 日常会主动使用的能力。

所以 FTS5 在这里的意义不是炫技术，而是让“历史可检索”真正足够便宜、足够实用。

---

## 5. 为什么 `session_search` 和 `lineage` 是一组设计

一旦你引入 compression，就会产生一个问题：

- 当前 session 里只剩摘要和最近几轮
- 很早的细节已经不在当前工作窗口里

这时候，`session_search` 的价值会立刻放大。

因为它可以：

- 沿着历史记录重新检索旧 session
- 找回压缩前更完整的表达
- 把旧版本里的细节重新带回当前上下文

如果没有 lineage，你搜到的可能只是“当前压缩版”；如果有 lineage，你更容易保留和搜索原始谱系节点。

所以：

- `h05` 解决“怎么压”
- `h06` 解决“怎么存”
- `h06a` 解决“怎么重新把存下来的历史用起来”

---

## 6. 什么时候 agent 应该主动用 `session_search`

通常有几类场景很适合：

### 场景 1：用户说“像上次那样做”

这类表达本身就要求 agent 去找过去的相似历史，而不是只靠当前窗口硬猜。

### 场景 2：当前任务和很早之前的会话有关

例如：

- 之前已经讨论过 provider fallback
- 之前已经确认过某套 skill 使用方式
- 之前已经写过类似 bridge doc 的结构

这时主动检索历史比重新从零推理更稳。

### 场景 3：上下文已经压缩过，当前窗口不再包含完整前情

这时候 `session_search` 就是在替 compression 补上一条“可回头再找”的路。

---

## 7. 为什么它不等于长期 memory

`session_search` 和 `memory` 很容易被混淆，但它们承担的是不同角色：

- `memory`：提炼过、相对稳定、适合长期注入 prompt 的知识
- `session_search`：保留原始历史、按需检索、临时拉回当前回合的过去记录

你可以把它们理解成：

- memory 像“长期整理过的笔记”
- session search 像“回档案柜里翻旧卷宗”

两者都重要，但不是同一种东西。

---

## 9. 这页最该带走的一句话

`session_search` 不是给数据库加一个查询接口，而是让 agent 在需要时能主动回到自己的历史里找线索，再把这些线索重新带回当前推理。

这就是为什么 Hermes 里的 session storage 最终会通向 agent-level tool，而不是停在存储层。
