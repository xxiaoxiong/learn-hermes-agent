# h11a — Command Registry Routing：为什么 slash command 的关键不是“命令名”，而是统一路由描述

> 这页是 `h11` 的延伸。主线章节已经介绍了 `COMMAND_REGISTRY`；这份 bridge doc 要把更深一层的理解讲清楚：**slash command 真正统一的不是字符串匹配，而是“同一份命令描述如何在多端被路由、解析、执行”。**

---

## 先说结论

`/search`、`/skills`、`/memory` 这些命令，看起来像是几条简单字符串。

但在 Hermes 里，它们真正重要的不是名字本身，而是背后那份统一的命令描述：

- 叫什么
- 有哪些别名
- 哪些平台支持
- 由谁处理
- 参数怎样解释

所以 `COMMAND_REGISTRY` 的价值，不是“有个字典存命令”，而是把**命令语义和平台入口解耦**。

---

## 1. 为什么“命令分发”问题比看起来更大

如果只做 CLI，一个 `if input.startswith("/search")` 也许就够了。

但 Hermes 面对的是：

- CLI
- Gateway
- Telegram
- 其他消息平台

它们的输入来源不同、输出格式不同、交互约束也不同。此时如果每个平台各写一套命令解析逻辑，系统很快就会变成：

- 命令名相同
- 实现细节不同
- 支持范围不一致
- 文档和行为开始漂移

所以真正的问题不是“怎么识别 `/search`”，而是：

> 怎么让一条命令在多个入口下仍然保持同一份定义。

---

## 2. registry 真正统一的是“命令对象”，不是“命令字符串”

这一点非常关键。

字符串只是入口；真正需要复用的是命令对象本身，包括：

- 主名和 alias
- 平台限制
- handler
- help 文案
- 参数 schema

也就是说，平台看到的不是“这串字符怎么处理”，而是：

> 这串字符最终解析到了哪一个 `CommandDef`。

这让命令系统从“到处散落的字符串判断”升级成了“统一命令模型驱动的路由系统”。

---

## 3. 为什么 alias 解析也不该是平台自己的事

看起来 alias 很像输入层小问题：

- `/s` 代表 `/search`
- `/skill` 可以通向 `/skills`

好像每个平台自己做掉也没什么。

但一旦这样做，就会产生微妙分裂：

- CLI 支持 `/s`
- Telegram 不支持 `/s`
- Gateway 又支持另一组简写

这时你维护的就不再是一套命令系统，而是多个长得像的命令方言。

所以 alias 必须属于公共解析层，而不是平台入口层。平台只负责把文本交出来，不负责重新定义命令语义。

---

## 4. 为什么 platform support 也应该写在命令描述里

有些命令可能只适合 CLI，有些命令适合所有端。

如果平台支持范围散落在各端实现里，结果会是：

- 文档不知道该写在哪支持
- 新平台接入时不知道哪些命令默认可用
- 同一命令在不同端的可用性越来越难追踪

把 `platforms` 放进 `CommandDef`，其实是在表达一件很重要的事：

> 平台支持性是命令定义的一部分，不是入口代码的偶然决定。

这会让整个路由层更可推导，也更容易做文档和 UI 展示。

---

## 5. 为什么这页和 Gateway 也直接相关

`h11` 看上去是 CLI 架构，但它和 `h12` Gateway 是天然连着的。

因为 gateway 并不是重新发明一套命令系统，而是复用同一份 registry：

- 平台入口把消息转成文本
- 文本送进公共 `resolve_command()`
- 得到统一 `CommandDef`
- 再由当前平台上下文决定如何格式化响应

所以命令系统和 gateway 的关系，不是“两个独立模块”，而是：

- `h11` 提供统一命令语义
- `h12` 提供多平台消息入口

二者拼起来，slash command 才真正跨端成立。

---

## 6. 一个很好用的判断法：这段逻辑是在解释“命令是什么”，还是在解释“平台如何收消息”

当你犹豫一段代码该放命令层还是平台层时，可以先问：

### 如果它是在解释命令是什么

例如：

- `/search` 有哪些 alias
- `/skills` 需要什么参数
- 哪些平台支持这条命令

那它属于 registry / resolve 层。

### 如果它是在解释平台如何收消息

例如：

- Telegram 消息文本怎么取
- CLI 输入怎么读
- Discord 交互事件怎么还原成命令字符串

那它属于入口适配层。

这个问题一问，边界就会清楚很多。

---

## 7. 为什么 skin engine 也能放进这张图里

看起来 `skin_engine.py` 只是 CLI 视觉层，和 command routing 没关系。

但它其实补上了一个重要事实：

- registry 决定“命令是什么”
- platform adapter 决定“消息从哪来”
- skin / rendering 决定“结果怎么被看见”

这说明 Hermes 在命令系统上并没有把“解析、执行、渲染”揉成一团，而是继续维持分层：

- 路由层
- 执行层
- 表现层

这也是为什么命令架构可以同时服务 CLI 和 Gateway，而不是绑死在终端交互里。

## 9. 这页最该带走的一句话

slash command 的核心不是“匹配到哪个命令名”，而是让所有平台都指向同一份命令描述对象，再由统一路由层决定怎么执行它。

这就是 `COMMAND_REGISTRY` 在 Hermes 里真正的价值。
