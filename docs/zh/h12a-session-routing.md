# h12a — Session Routing：为什么 Gateway 的真正难点不是接平台，而是隔离会话

> 这页是 `h12` 的延伸。主线章节已经介绍了 `GatewayRunner` 和 `MessageEvent`；这份 bridge doc 要把真正的架构难点点出来：**Gateway 的核心挑战不是把消息接进来，而是把每条消息稳定地路由到正确的会话上下文。**

---

## 先说结论

很多人第一次看 gateway，会把注意力放在：

- 怎么接 Telegram
- 怎么接 Discord
- 怎么处理 webhook / SDK

这些当然重要，但 Hermes 真正更关键的难点其实是：

- 哪条消息该进入哪个 session
- 同一个人跨平台是否共享上下文
- 同一个平台的不同 chat 是否应该隔离
- 自动任务、私聊、群聊是否应该复用同一 agent

也就是说，gateway 不是“把消息接进 agent”这么简单，它首先是一个**会话路由系统**。

---

## 1. 为什么接平台相对容易，而 session routing 更难

平台接入本质上还是适配器问题：

- 把平台原始事件转成统一结构
- 把回复再发回平台

这层虽然繁琐，但方向很明确。

真正更容易出错的是路由层，因为它关系到上下文污染：

- 本来应该独立的两段对话混在一起
- 本来应该连续的一段对话被拆开了
- 群聊和私聊不小心共用同一 session

一旦 routing 做错，agent 本身即使再聪明，也会在错误上下文里回答。

---

## 2. `platform + user + chat` 不是随便拼的，而是在定义上下文边界

`h12` 里看到的 session key 形状很像实现细节：

- `telegram:user:chat`
- `discord:user:channel`

但它其实在表达一句很重要的话：

> 上下文不是只属于“用户”，而是属于“用户在某个平台某个会话空间里的持续交互”。

这意味着：

- 同一个用户在 Telegram 和 Discord 不自动共享上下文
- 同一个用户在同一平台的不同群 / 私聊也不必共享上下文
- “谁在什么地方说的话”决定了上下文归属

所以 session key 不是拼接技巧，而是上下文隔离策略的编码形式。

---

## 3. 为什么只用 `user_id` 几乎一定会出错

只看 `user_id` 看起来最简单，但风险极高。

例如同一个用户：

- 在私聊里让 agent 帮自己写日报
- 在群聊里问一个公开问题

如果只按 `user_id` 路由，这两条线会被混到同一个 session 里。结果就是：

- 群聊上下文带进私聊
- 私聊上下文泄漏到群聊语境
- agent 开始出现“记得你刚才在另一个地方说过……”这种错位行为

所以 Gateway 的 routing 从来不是简单身份识别，而是上下文边界划分。

---

## 4. 为什么 DM pairing 也属于 routing 体系的一部分

表面上看，DM pairing 好像只是安全授权。

但它实际上也影响路由模型，因为它在回答：

- 哪些来源的消息应该被允许进入 agent loop
- 哪些 user_id 可以建立有效会话
- 某个平台上的私聊是否被视为可信上下文空间

也就是说，pairing 不只是“是否允许使用”，它同时也在决定：

> 哪些入口会被纳入正式 session routing 体系。

所以在 gateway 里，授权和路由并不是完全分开的两个世界。

---

## 5. 为什么这页和 cron / background tasks 也有关系

一旦你把 routing 看成“为每条事件找到正确上下文”，cron 就自然会进入同一张图。

因为 cron 任务也需要回答：

- 它属于哪个 session 空间
- 它是否应当总是 fresh agent
- 它和用户日常对话应否共享上下文

Hermes 对 cron 的回答通常是：

- fresh agent
- 独立上下文
- 不污染用户平时会话

这其实就是另一种 session routing 决策。

所以：

- gateway 处理外部实时事件的 routing
- cron 处理定时后台事件的 routing

两者共享的是同一种架构问题。

---

## 6. 一个很好用的判断法：这条事件更像“同一段对话的延续”，还是“另一段独立工作流”

当你犹豫某类输入该不该复用旧 session 时，可以先问：

### 如果它是同一段对话的延续

例如：

- 同一平台、同一 chat、同一用户连续发来的补充消息

那更适合复用原 session。

### 如果它是另一段独立工作流

例如：

- 不同平台来源
- 不同 chat 空间
- cron 触发的后台任务
- 需要 fresh 上下文的自动处理

那更适合新建或隔离 session。

这个判断比“是不是同一个用户”更靠谱得多。

---

## 7. 为什么 GatewayRunner 的真正价值是“让 AIAgent 不知道平台存在”

一旦 session routing 做对，AIAgent 就可以保持非常纯：

- 它只看文本和当前 session 上下文
- 它不关心消息来自 Telegram 还是 Discord
- 它不负责判断群聊、私聊、定时任务之间该不该共享历史

这说明 GatewayRunner 的本质不仅是适配器容器，更是：

**把平台差异和会话边界问题都挡在 agent 外面。**

这样 AIAgent 才能专注于 agent loop，而不是背负平台状态机。

## 9. 这页最该带走的一句话

Gateway 的核心不是把不同平台都接进来，而是确保每条进入系统的消息都能稳定落到正确的会话边界里。

平台适配只是入口，session routing 才是它真正的架构中心。
