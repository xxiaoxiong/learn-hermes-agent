# h09a — Approval Pipeline：为什么权限系统必须放在调度层而不是工具内部

> 这页是 `h09` 的延伸。主线章节已经给出了 deny → check → allow → ask 的四段 pipeline；这份 bridge doc 继续回答一个更底层的问题：**为什么 Hermes 要把权限判断放在工具调度层，而不是让每个工具自己决定什么算危险。**

---

## 先说结论

权限系统真正要统一的，不只是“危险命令列表”，而是：

- 检测时机
- 拦截顺序
- 放行 / 询问 / 拒绝的判定方式
- 审批结果如何进入后续执行链

这些东西一旦散落到每个工具内部，就几乎不可能再保持一致。

所以 Hermes 的关键设计不是“有 approval 逻辑”，而是：

**approval 逻辑位于调度层，处在所有工具真正执行之前。**

---

## 1. 为什么“每个工具自己检查”看起来很自然

从局部实现角度看，这种方式很有诱惑力：

- `bash` 工具自己检查危险 shell 命令
- `write_file` 工具自己检查敏感路径
- `network` 工具自己检查危险域名

好像每个工具最懂自己的风险。

但一旦系统变大，这个方案会迅速出现问题：

- 规则分散在不同文件里
- 相同风险在不同工具上的标准不一致
- 新工具容易忘记补权限逻辑
- 询问用户的体验也会变得不统一

也就是说，它局部上方便，系统上却会失控。

---

## 2. 调度层的意义：先判断能不能做，再决定由谁做

Hermes 把审批放在 dispatch 前，实际上是在坚持一个顺序：

1. 先看这次调用是否允许
2. 再决定真正交给哪个工具 handler 执行

这个顺序的好处在于：

- 所有工具共享同一个入口
- 所有危险检测共享同一个时机
- 审批结果不会依赖某个工具作者有没有想周全

换句话说，安全边界不由单个工具维护，而由系统统一维护。

---

## 3. 权限系统统一的不是“风险内容”，而是“决策流程”

很多人会把 approval 理解成一组危险正则。

但真正更重要的，是决策流程本身：

- 先硬拒绝哪些情况
- 哪些在 check mode 下直接拦截
- 哪些命中 allowlist 可以自动过
- 哪些需要 ask 用户

这就是为什么 `deny → check → allow → ask` 比“写几条危险命令规则”更重要。

因为规则可以一直加，但如果流程不统一，系统行为依然会不稳定。

---

## 4. 为什么 ask 必须是系统级行为，而不是工具私有行为

假设每个工具自己决定要不要向用户确认，会出现什么？

- `bash` 弹一种确认方式
- `write_file` 弹另一种提示文案
- `network` 根本不询问，直接放行

这样一来，用户面对的不是一套权限系统，而是多个风格混乱的小审批器。

Hermes 把 ask 设计成统一 callback，解决的就是这个问题：

- 谁来问，由环境决定（CLI / Gateway）
- 什么时候问，由统一 pipeline 决定
- 问完的结果如何进入执行，由调度层统一消费

所以 ask 不是某个工具的 UI，而是整个 agent runtime 的一部分。

---

## 5. 为什么 allowlist 也应该在调度层维护

如果 allowlist 放进工具内部，会产生两个麻烦：

- 同一条“安全规则”无法跨工具复用
- 你很难知道系统整体到底默认放过了哪些东西

放在调度层以后，allowlist 就不再是局部技巧，而是全局政策：

- 哪些操作默认可信
- 哪些操作必须走确认
- 哪些操作无论如何都不能过

这让权限系统从“工具局部判断”升级成了“运行时全局策略”。

---

## 6. 一个最好用的判断法：这条逻辑是在描述工具实现，还是在描述系统边界

当你在想一段权限代码该放哪时，可以先问：

### 如果它是在描述工具怎么工作

例如：

- `write_file` 怎样写入内容
- `bash` 怎样执行命令
- `network` 怎样发请求

那它属于工具内部。

### 如果它是在描述系统允许什么、不允许什么

例如：

- 哪些路径不可写
- 哪些命令要审批
- 哪些情况允许自动通过

那它更应该属于调度层。

这个问题一问，边界通常就清楚了。

---

## 7. 为什么这页和 `agent-level tool` 也有关

虽然 approval 处理的是普通工具调用，但它和 `h03a-agent-level-tools` 其实共享一个更大的主题：

> Hermes 很在意“控制面”和“执行面”的边界。

- agent-level tools 把内部状态能力留在控制面
- approval pipeline 把权限判断留在控制面
- 普通 handler 专注执行面

所以这页 bridge doc 真正想帮你看到的是：

Hermes 不是功能堆起来的系统，它在很多地方都刻意保留了一层“先控制、后执行”的架构分工。

---

## 8. 和主线章节怎么连起来

这页 bridge doc 最适合这样看：

- `h03`：先理解有些能力在 dispatch 前就要被拦截
- `h09`：再理解所有普通工具调用在真正执行前也要经过统一审批

这样你会更容易发现，Hermes 很多关键稳定性设计都不在 handler 内部，而在 handler 之前。

---

## 9. 这页最该带走的一句话

权限系统不是给工具“补一点安全检查”，而是要在所有工具执行之前，先由 runtime 统一决定这次调用是否应该发生。

这就是为什么 approval pipeline 必须位于调度层，而不是散落在各个工具内部。
