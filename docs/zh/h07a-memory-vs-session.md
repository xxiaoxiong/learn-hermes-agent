# h07a — Memory vs Session：为什么“存下完整历史”不等于“形成长期记忆”

> 这页是 `h07` 的延伸。主线章节已经说明了 `MEMORY.md` / `USER.md` 的作用；这份 bridge doc 要把另一个最容易混淆的边界讲清楚：**session storage 和 memory system 虽然都在“保存过去”，但它们保存的不是同一种东西。**

---

## 先说结论

如果只记一句话，这页最想让你记住的是：

- **session** 保存的是原始历史轨迹
- **memory** 保存的是提炼后的长期知识

前者偏“档案”，后者偏“笔记”。

这两者都重要，但不能互相替代。

---

## 1. 为什么很多人第一次都会把它们混在一起

直觉上看，`h06` 和 `h07` 都在做“把过去留下来”：

- `h06` 把 messages 落进 SQLite
- `h07` 把重要内容写进 `MEMORY.md` / `USER.md`

所以很容易形成一个误解：

> 既然 session 已经保存了完整历史，为什么还要 memory？

这个问题表面合理，实际上混淆了两件不同的事：

- 历史是否被保存
- 未来是否能高效地再次使用这段历史

---

## 2. Session 保存“发生过什么”，Memory 保存“以后应该记住什么”

Session 关注的是：

- 这个会话里说了什么
- 调了哪些工具
- 哪一步发生了什么结果
- 历史顺序是什么

Memory 关注的是：

- 这里面有没有值得长期保留的偏好或事实
- 下次开始时哪些知识应该直接进入 prompt
- 哪些内容不能每次都靠重新搜索才能想起来

所以你可以把它们理解成：

| 结构 | 更像什么 |
|---|---|
| Session | 原始工作录像 |
| Memory | 从录像里整理出的长期笔记 |

---

## 3. 为什么 Session 不能直接替代 Memory

如果没有 memory，理论上 agent 也许仍然能靠 `session_search` 去翻旧记录。

但这会带来几个问题：

### 问题 1：每次都要先检索

用户长期偏好如果每次都靠 search 才拿回来，成本和复杂度都更高。

### 问题 2：历史是原始的，不是提炼过的

Session 里保留的是当时完整上下文，不一定适合直接再次注入 prompt。

### 问题 3：压缩与谱系会让“直接读历史”更重

随着 session 变长、压缩次数变多，你能搜到历史，不代表适合每轮都把整段历史带回来。

所以 session 更像原材料，memory 更像经过筛选后的长期结论。

---

## 4. 为什么 Memory 也不能替代 Session

反过来也一样。

如果只保留 memory，而不保留 session，系统会马上失去很多能力：

- 无法追溯原始上下文
- 无法检查某条记忆最初是怎么得出的
- 无法用 FTS 去找当时讨论过的具体表达
- 无法恢复完整工具调用轨迹

也就是说：

- memory 适合高频复用
- session 适合回溯、检索、恢复、审计

memory 不是历史数据库，session 也不是长期认知摘要。

---

## 5. 一个最好用的判断问题：这条信息更像“原始记录”还是“长期原则”

当你犹豫某条内容该去哪时，可以先问：

### 如果它更像“原始记录”

例如：

- 本次工具调用的完整输出
- 某个会话里讨论过的具体上下文
- 某条错误信息的完整堆栈

那它更适合留在 session。

### 如果它更像“长期原则”

例如：

- 用户偏好简洁总结
- 当前项目默认用某个路径结构
- 某个长期协作习惯今后还会重复生效

那它更适合进入 memory。

---

## 6. 为什么 Hermes 要把两者做成两套结构，而不是一套大存储

把它们混在一起当然也能做出一个“统一大仓库”，但问题是使用方式完全不同：

| 维度 | Session | Memory |
|---|---|---|
| 读取方式 | 按需搜索 / 恢复 | 每轮直接注入 prompt |
| 数据粒度 | 原始、细节多 | 提炼、密度高 |
| 生命周期 | 可无限积累成谱系 | 需要去重和限长 |
| 主要用途 | 历史回溯 | 行为稳定化 |

正因为使用方式差异太大，混成一个结构反而会让语义和实现都变糊。

---

## 7. 这页和 `session_search` 的关系

一旦把 session 和 memory 边界看清，`session_search` 的位置也会更清楚：

- memory 负责把高价值长期信息直接带到眼前
- session_search 负责在需要时回历史档案里翻细节

所以两者不是竞争关系，而是分层协作：

- 能提前提炼的，写 memory
- 只在需要时才回去翻的，走 session search

---

## 8. 和主线章节怎么串起来

这页 bridge doc 最适合串着三章一起看：

- `h06`：你先理解 session 如何被持久化与搜索
- `h07`：你再理解长期记忆为什么要单独落地
- `h06a`：你再看到 session search 如何把历史调回来

这样就会形成一个完整闭环：

- 存档是 session
- 提炼是 memory
- 回忆是 session_search

---

## 9. 这页最该带走的一句话

Session system 解决的是“历史别丢”；memory system 解决的是“哪些东西以后不该再靠翻历史才能想起来”。

它们都在保存过去，但保存的是两种不同层级的过去。
