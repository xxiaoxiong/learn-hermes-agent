# h15a — Budget Sharing：为什么 subagent 的真正约束是共享预算，而不是并发数量

> 这页是 `h15` 的延伸。主线章节已经讲了 `IterationBudget` 和上下文隔离；这份 bridge doc 要把另一个经常被低估的点讲清楚：**subagent 系统真正防失控的关键，不是开几个线程，而是父子 agent 是否共享同一份预算。**

---

## 先说结论

很多人第一次看到 subagent delegation，会把重点放在：

- 能不能并发
- 开几个 worker
- 子 agent 跑得快不快

这些都重要，但 Hermes 真正更核心的约束其实是：

- 子 agent 是否消耗父 agent 的迭代预算
- 整个任务树是否受同一上限控制

因为如果预算不共享，再漂亮的并发实现也可能只是把失控扩散得更快。

---

## 1. 为什么“让每个子 agent 都有自己的预算”听起来合理

直觉上，这样似乎更独立：

- 父 agent 有 50 次迭代
- 子 agent A 再给 50 次
- 子 agent B 再给 50 次

看上去每个子任务都能充分发挥。

但问题也正是在这里：

- 父 agent 可以通过不断委派，把一次任务膨胀成多倍成本
- 每个子 agent 都可能继续再生子 agent
- 总迭代次数会失去全局上限

也就是说，不共享预算时，“委派”其实就成了绕过总资源限制的后门。

---

## 2. 共享预算本质上是在为“整个任务树”设上限

Hermes 的预算设计真正要限制的，不是某一个 agent，而是：

> 从父任务出发，整棵 delegation tree 一共最多能消耗多少次迭代。

这意味着：

- 父 agent 自己多走一步，要花预算
- 子 agent 多走一步，也要花同一池预算
- 再往下嵌套的孙 agent 仍然花同一池预算

所以 `IterationBudget` 不是实例级配额，而是任务树级配额。

这一点一旦想通，subagent 的资源治理逻辑就会非常清晰。

---

## 3. 为什么并发不是核心，隔离和约束才是核心

并发当然有价值：

- 多个子任务可以同时推进
- 总耗时更短

但并发本身不是 subagent 架构的灵魂。

Hermes 真正更在意的是两件事：

- **上下文隔离**：子任务有干净的工作空间
- **预算共享**：子任务不能绕开全局成本约束

如果只有并发而没有这两点，你得到的更像一批随手并行的 LLM 调用，而不是一个可治理的 subagent 系统。

---

## 4. 为什么共享预算和 context isolation 必须同时存在

这两点其实是一对：

- isolation 防止上下文互相污染
- budget sharing 防止资源无限扩张

如果只有 isolation，没有 budget sharing：

- 子 agent 很干净
- 但可以无限膨胀

如果只有 budget sharing，没有 isolation：

- 总成本被控制住了
- 但子任务和父任务会在同一堆 messages 里打架

所以 Hermes 的 delegation 不只是“spawn 子 agent”，而是同时回答两个问题：

- 子任务在哪个上下文里工作
- 子任务花的是谁的预算

---

## 5. 为什么错误处理也受预算模型影响

一旦子 agent 失败，父 agent 往往还能继续判断：

- 要不要重试
- 要不要跳过
- 要不要换策略

但这里有个前提：

- 所有这些补救动作仍然受同一预算池约束

也就是说，共享预算不仅限制“正常推进”，也限制“失败后的补救链”。

这让整个系统不会因为不断重试子任务而失控。

所以预算模型不仅是性能控制，也是错误恢复的边界条件。

---

## 6. 一个很好用的判断法：委派是在拆分工作，还是在绕开限制

当你设计某种 delegation 机制时，可以先问：

### 如果它是在拆分工作

- 子任务上下文独立
- 成本仍算在同一总账里
- 父任务要为委派行为负责

那这是健康 delegation。

### 如果它是在绕开限制

- 子任务获得新的独立预算池
- 递归委派后总成本无限膨胀
- 父任务表面只有 1 次调用，实际背后耗了很多倍资源

那这就不是 delegation，而是资源逃逸。

这个问题几乎可以直接判定架构是否稳。

---

## 7. 为什么这页和 error recovery / cron 也会连起来

只要你把 budget sharing 看成“任务树总约束”，很多章节都会和它连上：

- cron job 里如果触发 subagent，也必须服从同一任务预算思路
- error recovery 里的 retry / continuation 也会继续消耗预算
- provider fallback 并没有绕开预算，只是换了一条执行路径

这说明预算不是 delegation 私有概念，而是整个 agent runtime 的通用约束语言。

## 9. 这页最该带走的一句话

Subagent 系统真正防止失控的关键，不是限制能开多少个并发 worker，而是确保所有父子 agent 共享同一份任务预算。

上下文可以隔离，但成本不能逃逸——这就是 Hermes delegation 设计里最关键的那条线。
