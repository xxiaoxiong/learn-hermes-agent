# h05a — Lineage Model：为什么压缩后的 session 不能覆盖原历史

> 这页是 `h05` 的延伸。主线章节已经告诉你 context compression 会生成新的 `lineage_id`；这份 bridge doc 要把背后的设计讲透：**为什么压缩不是覆盖旧 session，而是生成一条新的谱系节点。**

---

## 先说结论

如果压缩直接覆盖旧 session，系统会立刻失去三个重要能力：

- 无法追溯压缩前的完整历史
- 无法解释“当前摘要是从哪段历史提炼出来的”
- 无法让搜索系统在原始版本和压缩版本之间自由跳转

所以 Hermes 的思路不是：

> 把旧历史改短一点

而是：

> 生成一个新的、较短的 session，并让它指向之前的父 session

这就是 lineage model 的核心。

---

## 1. 为什么“覆盖旧 session”很诱人

表面上看，直接覆盖是最省事的：

- 数据库只保留一个 session id
- 当前上下文变短了
- 存储空间看起来也更省

但这个想法的问题在于，它把“当前工作窗口”和“历史事实档案”混成了一回事。

压缩要解决的是：

- 当前窗口太大，不方便继续推理

它不应该顺手把：

- 历史曾经发生过什么

也一起改写掉。

---

## 2. 压缩前后的两个 session，不是同一个语义对象

压缩前的 session 代表的是：

- 原始消息轨迹
- 完整的 tool call / tool result 历史
- 真实发生过的上下文序列

压缩后的 session 代表的是：

- 为了继续运行而重组过的工作窗口
- 中间部分已经被摘要折叠
- 更适合下一轮推理，但不再等于原始历史

这两个对象都真实存在，但它们承担的职责不同。

所以最合理的做法不是覆盖，而是并存。

---

## 3. lineage 的本质：让“可运行窗口”和“历史档案”同时成立

你可以把 lineage 理解成一条谱系链：

```text
session-001  原始完整历史
   ↓ compress
session-002  压缩后的工作窗口
   ↓ compress again
session-003  再次压缩后的工作窗口
```

这里最重要的不是编号，而是父子关系：

- `session-002.parent_session_id = session-001`
- `session-003.parent_session_id = session-002`

这样系统就可以同时拥有两件事：

- 当前够短、够干净、适合继续跑
- 原始历史仍然可追溯、可检索、可解释

---

## 4. 为什么摘要块本身不能替代原历史

很多人会想：

> 既然已经有 summary 了，原始历史是不是就不重要了？

这正是最容易踩的误区。

因为 summary 的语义地位和原始历史完全不同：

- summary 是压缩后的解释层
- 原始 history 是发生过的事实层

摘要可以：

- 帮助下一轮模型快速理解前情
- 降低上下文长度

但摘要不能：

- 保证保留所有细节
- 作为审计级别的原始记录
- 替代以后做更细粒度搜索的基础数据

所以 summary 是运行时便利，不是历史真相的替身。

---

## 5. lineage 和 FTS / session search 为什么天然联动

一旦你把 session 设计成有谱系，很多后续能力就顺了：

- FTS5 可以索引原始 session
- 当前运行窗口可以引用压缩后的 session
- `session_search` 可以搜到旧版本，再把结果带回当前上下文

如果没有 lineage，而只有“被覆盖后的当前版本”，那很多事情都会变得模糊：

- 搜到的到底是原始说法，还是摘要改写后的版本？
- 当前 session 的内容，是否还对应最初的真实工具轨迹？
- 某个关键信息是原本就存在，还是压缩时被抽象概括出来的？

lineage 正是在给这些问题保留答案。

---

## 6. 为什么这不只是数据库实现细节

很多人会把 `parent_session_id` 看成存储层小技巧。

其实不是。

它会反过来影响你对整个 agent 系统的理解：

- session 不是单个静态盒子，而是一条会演化的谱系
- compression 不是 destructive rewrite，而是派生新节点
- 搜索、恢复、审计看到的都不只是“当前状态”，而是“状态如何演化而来”

也就是说，lineage model 让会话从“快照”升级成了“可追踪的历史链”。

---

## 7. 什么时候你会明显感觉到 lineage 设计的价值

下面几种场景里，lineage 的价值会特别直观：

### 场景 1：用户说“你之前不是这么说的”

如果历史被覆盖，你只能看到压缩后的摘要版本；如果有 lineage，你可以顺着父链回去看原始上下文。

### 场景 2：你要搜索很早之前的一段关键词

如果原始 session 还在，FTS 就能找到；如果只剩摘要版，很多细词可能已经丢失。

### 场景 3：你要调试压缩策略是不是把重要信息折掉了

有 lineage 时，你可以把压缩前后两个 session 对比；没有 lineage 时，你根本没有可比的原始版本。

## 9. 这页最该带走的一句话

压缩解决的是“当前窗口太长”，不是“历史不再重要”。

所以 Hermes 的 lineage model 不是可有可无的附加层，而是保证压缩、搜索、恢复、审计能同时成立的核心结构。
