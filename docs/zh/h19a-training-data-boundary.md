# h19a — Training Data Boundary：为什么轨迹收集的难点不是“能不能存下来”，而是“什么值得进入训练集”

> 这页是 `h19` 的延伸。主线章节已经讲了 trajectory recorder、batch runner 和 ShareGPT 格式；这份 bridge doc 要把训练数据视角下最关键的那条边界讲清楚：**agent 运行日志不等于训练样本，真正困难的是决定哪些轨迹值得被保留、过滤、压缩和学习。**

---

## 先说结论

很多人第一次看到 trajectory saving，会自然觉得：

- 只要把每次 agent 对话都存下来
- 数据量够大
- 之后训练时再统一处理

但真实情况是，日志能不能存下来，往往不是最难的问题。

更难的问题其实是：

- 哪些轨迹应该进训练集
- 哪些只是调试日志
- 哪些应该直接丢掉
- 哪些要先压缩、清洗、脱敏、重写格式

也就是说，`h19` 真正的难点从来不是 persistence，
而是 dataset curation。

---

## 1. 为什么“全部保存”并不等于“全部可训练”

一次 agent 运行里，会出现很多不同性质的信息：

- 用户真实需求
- 模型推理与工具调用
- 工具原始回传结果
- 失败重试路径
- 辅助性内部动作
- 临时噪声与无效循环

如果你把这些一股脑全部灌进训练集，通常不会让模型更强，
反而可能让它学到很多你并不想保留的行为模式。

例如：

- 失控循环
- 冗长低效的工具使用
- 过度依赖内部辅助工具
- 暴露系统实现细节的中间产物

所以“能保存”只是日志层成功，
离“值得训练”还差很远。

---

## 2. 为什么 trajectory 和 dataset 之间必须有一道过滤边界

Hermes 在做 trajectory 保存时，其实天然存在两层目标：

### 日志目标

- 方便调试
- 方便复盘
- 方便审计失败路径

### 训练目标

- 给模型提供可学习的行为模式
- 尽量保留高质量、高信号样本
- 减少噪声和错误诱导

这两层目标并不总一致。

一个对调试很有价值的失败轨迹，
对训练却可能是负样本，甚至是脏样本。

所以 trajectory collection 和 training dataset 之间，必须有一道明确的过滤边界。

---

## 3. 为什么格式统一只是第一步，不是最后一步

把轨迹转成 ShareGPT 或其他统一格式当然很重要，
因为没有统一格式，后续数据处理会非常混乱。

但格式统一只解决了“能不能处理”的问题，
并没有解决“该不该保留”的问题。

也就是说：

- `to_sharegpt()` 解决的是表示问题
- `TrajectoryFilter` 解决的是样本质量问题

如果把格式转换误当成数据清洗本身，就会产生一种错觉：

> 只要导出成标准格式，数据就已经可用了。

真实情况通常完全不是这样。

---

## 4. 为什么低质量轨迹的危害往往比“数据少一点”更大

初学者常见直觉是：

- 先多收集，再说

但训练数据和普通日志不一样。

低质量样本不是“没帮助而已”，
它往往会真的把模型往坏方向拉：

- 学到无意义循环
- 学到错误的工具调用顺序
- 学到对失败路径的错误偏好
- 学到大量 Hermes 内部实现细节，而不是任务完成能力

所以训练数据里最怕的不是少，
而是“噪声很大但表面看起来很多”。

在很多场景下，少而干净的轨迹会比多而混乱的轨迹好得多。

---

## 5. 为什么内部辅助动作不一定应该直接进入训练集

Hermes 运行里会有一些内部性很强的动作，例如：

- memory write
- session search
- 某些 runtime-specific routing 痕迹
- 调试和恢复用的中间元数据

这些动作对 Hermes 自身运行当然很重要，
但它们不一定等同于“应该让模型学会的行为”。

这里就出现了一条非常关键的边界：

> 哪些是完成任务所需的通用 agent 行为，哪些只是特定系统里的内部实现细节？

如果这条线不先划清，模型最后学到的可能不是“如何解决任务”，
而是“如何模仿某个运行时的表层痕迹”。

---

## 6. 为什么 reward / filter / export 本质上是同一条线上的不同关卡

在 `h19` 主线里你会看到：

- 轨迹收集
- 质量过滤
- reward 计算
- 导出训练样本

它们看起来像几段不同流程，
但如果从训练边界角度看，其实都在回答同一个问题：

> 这一段运行，值不值得被模型学习？

区别只在于：

- filter 更像硬性门槛
- reward 更像质量评分
- export 决定最终进入哪种训练格式

所以这些步骤并不是松散拼起来的流水线，
而是围绕“保留什么样本”这一核心判断层层收紧。

---

## 7. 为什么失败轨迹也不是一律无用

说到这里又容易走到另一个极端：

- 那失败轨迹是不是都应该丢掉？

也不一定。

关键不在于“成功 or 失败”这个单一标签，
而在于这条轨迹是否携带了对训练有价值的信号。

例如：

- 某些失败轨迹适合做调试数据
- 某些失败轨迹适合做负反馈学习
- 某些失败轨迹则只是纯噪声，应直接丢弃

所以真正需要的是样本分层，而不是简单二分。

这一点也解释了为什么 `trajectory_samples.jsonl` 和 `failed_trajectories.jsonl` 分流保存很有意义：

- 它们首先在数据层被区分
- 后面才有可能在训练策略层做不同处理

---

## 8. 一个很好用的判断法

当你面对一类 trajectory 时，可以先问四个问题：

### 它是否完成了清晰任务？

如果任务目标本身就模糊，训练价值通常不高。

### 它是否展示了可取的 agent 行为模式？

例如高效工具使用、合理停止、清晰总结。

### 它是否夹带过多系统内部噪声？

如果主要可见内容都是 runtime-specific 中间动作，就要谨慎。

### 它是否需要先压缩、过滤、脱敏后才能进入训练集？

很多样本不是不能用，而是不能直接用。

这个判断法很简单，但能帮你避免“日志即样本”的误区。

## 10. 这页最该带走的一句话

Agent 运行日志不等于训练样本。

真正困难的地方不是把轨迹存下来，而是持续判断：哪些轨迹值得进入训练集，哪些只适合调试，哪些应该被清洗后再学习，哪些应该直接被丢掉。
