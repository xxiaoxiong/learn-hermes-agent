# h14a — Hook Boundary：为什么 hook 能扩展行为，却不能接管主循环

> 这页是 `h14` 的延伸。主线章节已经强调 hook 只能观察和注解；这份 bridge doc 要把这个限制讲得更彻底：**hook 的价值恰恰建立在它“不能接管控制流”这条边界之上。**

---

## 先说结论

很多人一看到 hook，就会自然联想到：

- 那我是不是可以在这里改执行顺序？
- 能不能直接阻止某个工具？
- 能不能让插件在这里替换主循环逻辑？

这些冲动很正常，但 Hermes 恰恰在有意识地拒绝它。

因为一旦 hook 既能观察又能控制，系统就会很快失去可预测性。

所以 hook 的真正定位是：

**围绕主循环注入附加行为，而不是接管主循环本身。**

---

## 1. 为什么“让 hook 更强大一点”听起来总是很诱人

从插件作者角度看，最理想的世界似乎是：

- 想在 pre hook 里直接改参数
- 想在 post hook 里替换结果
- 想在 error hook 里改变下一步控制流

这样插件看起来无所不能。

但代价也会立刻出现：

- 主循环再也不是唯一控制中心
- 不同 hook 之间会相互影响甚至打架
- 某个插件是否注册，都会改变系统核心语义

这会让“可扩展”迅速滑向“不可预测”。

---

## 2. hook 和 approval / agent-level tool 的边界必须分清

Hermes 里已经有几种真正参与控制流的机制：

- agent-level tool 拦截
- approval pipeline
- 主循环自己的 dispatch / continuation 逻辑

这些模块之所以存在，就是为了让控制面有明确归属。

如果 hook 也开始接管这些事情，就会出现边界重叠：

- 某些插件在 pre tool hook 里偷偷做 deny
- 某些插件在 post hook 里重写 tool result
- 某些插件试图绕过 approval 规则

到最后，谁才真正控制 agent，就不再清楚。

所以 Hermes 保留 hook 的“弱权力”，反而是在保护系统整体的可解释性。

---

## 3. 为什么“只能观察和注解”并不意味着 hook 很弱

很多人一听不能控制流，就会误判 hook 没什么用。

其实不是。

只要你把它放在正确位置，它仍然非常强：

- 记录日志
- 统计耗时
- 上报遥测
- 注入观测数据
- 触发通知
- 在错误发生时做旁路记录

这些都很重要，而且它们有一个共同特点：

**它们增强系统的可见性和可集成性，但不改写系统的核心执行语义。**

这正是健康 hook 系统最该做的事。

---

## 4. 为什么 hook 异常不能影响主循环

`h14` 里有一个非常关键但很容易被低估的设计：

- hook 自己报错，不该拖垮 agent 主流程

这不是单纯的容错技巧，而是在表达 hook 的地位：

- hook 是附加层
- 主循环是基础层

附加层失败，不能让基础层跟着倒。

否则你就会得到一个很脆弱的系统：

- 只是某个统计插件挂了
- 结果整个 agent 无法继续工作

这显然违背了 hook 设计的初衷。

---

## 5. 一个很好用的判断法：这段扩展逻辑是在“描述过程”，还是在“改变决策”

当你犹豫某个插件能力该不该放进 hook 时，可以先问：

### 如果它是在描述过程

例如：

- 记录某次 tool call 花了多久
- 在 response 后发一条通知
- 给 observability 系统发送 telemetry

那很适合 hook。

### 如果它是在改变决策

例如：

- 阻止某个工具被执行
- 改写工具参数
- 决定是否继续下一轮循环

那它通常不该由 hook 承担，而应进入 approval、dispatch、agent-level tool 或主循环本身。

这个问题几乎能一刀切开边界。

---

## 6. 为什么 plugin 可扩展性反而依赖 hook 的克制

这点有点反直觉，但非常重要：

> 插件系统之所以能安全扩展，恰恰因为 hook 没有被赋予过大的控制权。

如果 plugin 能随意改变控制流，那么：

- 两个插件组合后可能出现不可预测行为
- 用户很难知道系统当前到底遵循哪套逻辑
- 升级、排障、复现问题都会变得极难

而当 hook 保持在观察 / 注解边界内，插件就更容易：

- 叠加使用
- 独立排查
- 可控卸载

所以“克制”不是限制插件，而是在为插件生态保留秩序。

---

## 7. 为什么这页和 `h18 Plugin System` 是强绑定的

如果不先把 hook boundary 看清，后面看 plugin system 时就很容易期待过高：

- 以为插件能像 fork 一样接管 Hermes
- 以为 register_hook 就等于拥有主循环控制权
- 以为任何 runtime 政策都可以放插件里改

但 Hermes 的 plugin 更准确的定位是：

- 注册工具
- 注册 hook
- 注册命令
- 扩展外围能力

而不是重写 agent 核心调度。

所以这页其实是在为 `h18` 提前打地基。

---

## 9. 这页最该带走的一句话

hook 的价值不在于它能偷偷改写主循环，而在于它能在不改写主循环的前提下，为系统增加观察、集成和附加行为。

正因为 Hermes 守住了这条边界，plugin 才不会把 agent 变成一团不可预测的分支逻辑。
